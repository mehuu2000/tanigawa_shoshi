## 概要

本システムは、最終的には書誌同定のために以下の流れで処理を行う。

1. 書誌データのトークン化（アプリ側）
2. Solrへの登録（フィールド別インデックス）
3. 参考文献文字列の検索
4. 候補文献の取得
5. 再ランキング（RC / CC / MC）
6. 同定判定

今回の実装範囲は **1〜4のSolr登録・インデックス作成・簡単な検索まで** とする。
5〜6のRC / CC / MCによるスコアリングと閾値判定は後続フェーズで扱う。

本実装では、**トークン生成を完全にアプリ側で制御する方式**を採用する。

Solrへの登録処理やスキーマ設定は、既存の `../jalc/jalc-to-solr.ipynb` の考え方を参考にする。
ただし、コードをそのまま流用するのではなく、本リポジトリ側で独自に実装する。
論文再現のために必要なフィールド構成・トークン生成・検索対象は、本実装の方針に合わせて定義する。

実装時は、ノートブックに処理を直書きしすぎず、独立した処理を関数・モジュールに分ける。
特に、MongoDBからの抽出、JaLC文書の整形、トークン化、Solrスキーマ設定、Solr登録、検索処理は責務を分けて管理する。

## 1. トークン化方針（アプリ側）

### 基本方針

書誌データおよび入力文字列に対して、同一ロジックでトークン化を行う。

### 処理手順

1. 文字列をICU等で大まかに分割
2. 各要素が日本語かどうかを判定
3. 日本語連続列 / 非日本語連続列に再構成
4. 各タイプに応じてトークン生成

### トークン生成ルール

* 日本語（japanese）

  * 文字2-gram

* 日本語以外（non_japanese_seq）

  * 単語1-gram
  * 隣接単語2-gram

### 注意

* クエリ側とインデックス側で**完全に同一ロジック**を使用する
* トークンは集合として扱う（順序・重複は考慮しない）

---

## 2. Solrへの登録

登録対象は、MongoDB `jalc.restapi` のうち `content_type = "JA"` の文献に限定する。

### フィールド構成

各書誌データについて以下を保持する：

#### 保存フィールド（表示・再計算用）

* authors
* first_author
* title
* journal
* year
* volume
* page

設定：

* `stored=true`
* `indexed=false`

保存フィールドは、複数表記を保持できるよう配列として登録する。
保存フィールド名は論文の書誌要素名に合わせ、`*_raw` とはしない。
著者名・論文タイトル・雑誌名は、言語表記や略称などのバリエーションを配列内に保持し、後続のスコアリング時に `C_f^v` を構築できるようにする。
`first_author` は後続のCC計算用に保存する。
`volume` は論文の「巻・号フィールド」に対応させるため、MongoDBの `volume` と `issue` を同じ `volume` 配列内に分けて保持する。

#### トークンフィールド（検索用）

* authors_tokens
* first_author_tokens
* title_tokens
* journal_tokens
* year_tokens
* volume_tokens
* page_tokens

設定：

* `stored=false`
* `indexed=true`

※ 検索用トークンはインデックス専用とし、ストレージ容量を抑えるため `stored=false` とする。
トークン内容の確認は、登録処理時のログやローカルの tokenize() 実行結果で行う。

※ all_tokens は使用しない

各トークンフィールドは、対応する保存フィールド配列の各要素をトークン化し、重複除去した集合として登録する。
`volume_tokens` は `volume` に含めた巻と号の両方から生成する。

## 3. 検索処理

### 入力

* 参考文献文字列

### 処理

1. 入力文字列をアプリ側でトークン化
2. 複数トークンフィールドを対象に全文検索を行う

今回の検索対象は `authors_tokens`, `title_tokens`, `journal_tokens`, `year_tokens`, `volume_tokens`, `page_tokens` とする。
`first_author_tokens` は登録するが、検索対象には含めない。

### 検索の考え方

論文における

```
C = ⋃ Cf
```

に対応し、複数フィールドを**統合集合として扱う検索**を行う。

実装上は複数フィールド検索（edismax等）で実現する。

---

## 4. 候補文献の取得

* SolrのBM25によりランキングされた結果を取得
* 基本は**上位1件（第一位）を対象とする**
* 実装上は上位K件取得してもよい（デバッグ・拡張用）

---

## 5. 後続フェーズ：再ランキング（RC / CC / MC）

今回の実装ではここは実装しない。
Solrには、後続フェーズで再トークン化・スコア計算できるよう保存フィールドを保持する。

取得した候補文献に対して、再度アプリ側でトークン化を行いスコアを計算する。

### 使用フィールド

#### RC（Reference Coverage）

```
F = {authors, title, journal, year, volume, page}
```

#### CC（Citation Coverage）

```
F' = {first author, title, journal, year, volume, page}
```

※ CCでは authors 全体ではなく **first author のみ使用**

---

### 計算

* RC：トークン集合の一致率
* CC：フィールドごとの一致率の平均
* MC：RCとCCの調和平均

---

## 6. 後続フェーズ：同定判定

今回の実装ではここは実装しない。

* 第一位候補文献のスコア（RC / CC / MC）を評価
* 閾値以上であれば同定成功とする

## 設計の特徴

* トークン生成を完全にアプリ側で制御
* 論文のトークン定義を厳密に再現可能
* 今回は検索をSolrで行い、後続フェーズの意味評価はアプリ側で分離
* 既存 `jalc-to-solr.ipynb` のJaLC抽出・Solr登録方針を参考にしつつ、本リポジトリで独自実装する

## 実装構成方針

処理はできるだけ関数として分け、独立性の高い処理はフォルダ・モジュールを分ける。

想定する責務分割：

* MongoDB接続・対象文献取得
* JaLC文書から保存フィールドへの変換
* 論文準拠トークン化
* Solrスキーマ設定
* Solrへのバッチ登録
* 参考文献文字列の検索
