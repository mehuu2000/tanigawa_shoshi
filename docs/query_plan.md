# Query Plan（検索処理設計）

## 概要

検索時（参考文献文字列入力）も、インデックス作成時と**完全に同一のトークン化ルール**を適用する。

検索処理は、トークン化処理とSolr検索処理を分けて実装する。
Solr接続や検索パラメータの扱いは既存の `../jalc/jalc-to-solr.ipynb` の利用方法を参考にするが、実装は本リポジトリ内で新しく行い、関数化して再利用しやすくする。

---

## トークン生成ルール

* 日本語：文字2-gram
* 日本語以外（英語・数字など）：単語1-gram
* 日本語以外が連続する場合：単語2-gramを追加

---

```text
JaLC Reference Coverage 2026年創業
```

## トークンの扱い

* トークンは集合として扱う
* 順序は考慮しない
* 重複は除去する

---

## 参考文献文字列の分割

参考文献文字列は MongoDB の書誌フィールドとは表示形式が異なり、`,`、`:`、`[]`、`()` などの区切り記号で要素が連結されていることが多い。
検索時は、これらの区切り記号で参考文献文字列を疑似フィールド配列に分割してから、各要素に `tokenize_values()` を適用する。

これにより、区切り記号をまたいだ不要な非日本語2-gramを作らず、MongoDBから抽出した保存フィールド配列に近い単位でクエリトークンを生成する。

---

## 検索対象フィールド

```text
all_tokens
```

`all_tokens` は `authors`, `title`, `journal`, `year`, `volume`, `page` から生成したトークンを統合した検索用フィールド。
`first_author` はSolrに stored field として登録するが、`all_tokens` の生成元には含めない。
後続のCC計算で使用するためのフィールドとして扱う。

---

## 検索の考え方（重要）

論文では

```text
C = ⋃ Cf
```

として、全フィールドのトークン集合を検索対象としている。

本実装では、登録時に `all_tokens` を作成して `C` を再現する。
検索時は `all_tokens` 単一フィールドに対して BM25 による候補取得を行う。

---

## クエリ生成

```python
reference_values = split_reference_values(reference_string)
tokens = tokenize_values(reference_values)
query_str = " ".join(tokens)

params = {
    "q": query_str,
    "df": "all_tokens"
}
```

---

## 検索の意味

このクエリは以下を意味する：

```text
tokens ∈ all_tokens
```

つまり、

👉 **登録時に作成した全フィールドの統合集合 C に対する検索**

---

## 検索結果

* BM25によりスコアリング
* 上位K件を取得
* 今回の実装では候補取得までを対象とする
* 論文準拠の第一位候補評価は後続フェーズで扱う

---

## 全体フロー

```text
参考文献文字列
    ↓
トークン化（アプリ）
    ↓
トークン集合生成
    ↓
all_tokens 検索（C = ⋃ Cf）
    ↓
候補取得（BM25）
```

RC / CC / MCによる再ランキングと閾値判定は後続フェーズで扱う。

---

## 注意（最重要）

インデックス生成時と検索時でトークン化ルールが異なる場合、検索精度は大きく低下する。

👉 必ず同一ロジックを使用すること

---

## 設計意図

### なぜ単語2-gramを入れるのか

* フレーズ一致を強化
* 誤マッチを減らす
* BM25のスコアを適切に反映

---

## 結論

* 検索は `all_tokens` 単一フィールドに対して行う
* トークンはアプリ側で完全制御する
* Solrは候補取得に専念させる

👉 **今回の検索は「集合C」に対するSolr/BM25の候補取得までを行う**

RC / CC / MCによる評価は後続フェーズで実装する。
