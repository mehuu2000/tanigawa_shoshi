# Evaluation Plan（再ランキング・閾値評価設計）

## 目的

本フェーズでは、現在の Solr 検索実装を候補取得エンジンとして使い、その後段でアプリ側のスコアリングと閾値判定を行う。

対象とする処理は以下の通り。

1. 参考文献文字列をクエリとして Solr 検索を行う
2. BM25 上位 100 件を候補として取得する
3. 候補ごとに RC / CC / MC を計算する
4. RC 順位、CC 順位、MC 順位をそれぞれ作る
5. 各スコアリング方式の 1 位候補に対して閾値を適用する
6. 正例・負例データに対して評価を行う

---

## 本実装の立場

`docs/first_manuscript_tanigawa.pdf` は方針の参考にするが、本リポジトリのシステムは論文の実装そのものではない。

特に以下は、本実装独自の前提である。

* 候補取得は現在の Solr / BM25 をそのまま使う
* BM25 上位 100 件に正解候補が含まれることを前提に、後段で RC / CC / MC により再ランキングする
* 閾値は論文記載値をそのまま使う

このため、本フェーズは「論文の完全再現」ではなく、「現在の Solr 実装を前段に置いた実用的な再ランキング・評価系」として設計する。

## 候補取得方針

### Solr の役割

Solr は候補取得専用とする。

* 入力: 参考文献文字列
* 検索対象: `all_tokens`
* スコアリング: Solr 標準の BM25
* 出力: 上位 100 件

### 上位 100 件を使う理由

上位 100 件を使う理由は以下の通り。

* BM25 の上位であれば正解候補が含まれている可能性が高い
* 全件に RC / CC / MC を計算するより計算コストを抑えられる
* 候補が少なすぎると後段の再ランキングで正解を救えない可能性がある

本値は実装上の設計値であり、後から変更可能とする。

## スコアリング定義

### 使用するトークン集合

クエリ側トークン集合:

```text
R = 参考文献文字列から生成したトークン集合
```

候補文献側トークン集合:

```text
C = ⋃_{f ∈ F} C_f
```

ここで、

```text
F = {authors, title, journal, year, volume, page}
```

である。

CC 用のフィールド集合は以下とする。

```text
F' = {first_author, title, journal, year, volume, page}
```

### RC

RC はクエリ側トークンの被覆率とする。

```text
RC = |R ∩ C| / |R|
```

### CC

CC は候補文献の各フィールドの被覆率の平均とする。

```text
CC = (1 / |F'|) * Σ_{f ∈ F'} CCf
```

各フィールドの被覆率 `CCf` は以下で定義する。

* `first_author`, `title`, `journal`:
  保存されている各表記バリエーションごとにトークン集合を作り、その中で最大値を採用する
* `year`, `volume`, `page`:
  単一フィールド集合としてそのまま使う

```text
CCf = max_v (|R ∩ Cvf| / |Cvf|)
```

CC の各フィールドと保存フィールドの対応は以下の通り。

* `first_author`:
  `first_author_variations` の各要素をそれぞれ 1 つの表記バリエーションとして扱い、各要素を個別にトークン化して `Cvf` を作る
* `title`:
  `title_variations` の各要素をそれぞれ 1 つの表記バリエーションとして扱い、各要素を個別にトークン化して `Cvf` を作る
* `journal`:
  `journal` 配列の各要素をそれぞれ 1 つの表記バリエーションとして扱い、各要素を個別にトークン化して `Cvf` を作る
* `year`, `volume`, `page`:
  配列全体から 1 つのトークン集合 `C_f` を作る

`authors_variations` は CC では使わない。
CC の著者系フィールドは `authors` ではなく `first_author` を使うためである。

実装上は、`year`, `volume`, `page` も「要素 1 個だけのバリエーション列」とみなして扱ってよい。
つまり、これらのフィールドも `max_v` を使う共通ロジックにそのまま載せ、候補となるトークン集合が 1 個だけ入った配列として計算してよい。

### MC

MC は RC と CC の調和平均とする。

```text
MC = 2 * RC * CC / (RC + CC)
```

ただし `RC = 0` かつ `CC = 0` の場合は `MC = 0` とする。

## 再トークン化方針

候補文献の RC / CC / MC 計算には、Solr に保存してある生データをアプリ側で再トークン化して使う。

使う保存フィールドは以下の通り。

* `authors`
* `first_author`
* `title`
* `authors_variations`
* `first_author_variations`
* `title_variations`
* `journal`
* `year`
* `volume`
* `page`

方針:

* `all_tokens` は候補取得専用とし、後段評価では使わない
* トークナイザは登録時・検索時・評価時で完全一致させる
* トークンは集合として扱い、順序や重複は考慮しない
* まず RC / CC で共通利用するフィールド別統合集合 `Cf` を作る
* RC では必要な `Cf` を統合して候補文献全体の統合トークン集合 `C = ⋃_{f ∈ F} C_f` を作る
* CC 用には、表記バリエーションごとのトークン集合列 `Cvf` を追加で作る

つまり、候補文献の再トークン化結果は 1 種類ではなく、少なくとも以下の 2 系統を持つ。

* 共通の中間表現:
  フィールドごとに統合したトークン集合 `Cf`
* CC 用:
  各表記バリエーションを個別にトークン化したトークン集合の列

RC の計算時は、まず `Cf` を使って必要なフィールド集合を選び、それらを統合して `C` を作る。
CC の計算時は、`first_author`, `title`, `journal` では `Cvf` を使い、`year`, `volume`, `page` では `Cf` を使う。

## リランキング方針

上位 100 件に対して以下を行う。

1. 各候補の RC を計算
2. 各候補の CC を計算
3. 各候補の MC を計算
4. RC 順位、CC 順位、MC 順位を別々に作る

重要なのは、RC / CC / MC はそれぞれ独立のランキングになる点である。

したがって、最終判定も以下の 3 系統で別々に扱う。

* RC 方式
* CC 方式
* MC 方式

同点時の優先順は、実装上は BM25 順位を使う方針を推奨する。

なお、`MC` は `RC` と `CC` の調和平均であるため、実装上は各候補について先に計算した `RC` と `CC` の値を再利用して求める。
`MC` の計算時に `RC` / `CC` を再度計算する必要はない。
ここで「RC / CC / MC は独立のランキング」とは、順位付けと閾値判定を方式ごとに別々に行う、という意味である。

---

## 閾値判定方針

各方式について、その方式で 1 位になった候補に対してのみ閾値を適用する。

* RC 順位 1 位候補に RC 閾値を適用
* CC 順位 1 位候補に CC 閾値を適用
* MC 順位 1 位候補に MC 閾値を適用

判定:

* 閾値以上: 同定成功
* 閾値未満: 同定結果なし

本実装では、閾値は論文記載値をそのまま使う。

```text
RC = 0.9375
CC = 0.8730
MC = 0.8623
```

本システムは候補取得方法が論文実装と完全一致しているわけではないが、まずは比較基準を固定するため、この値をそのまま適用する。

## 評価データ方針

### 前提

正例データ、負例データはリポジトリ内には存在しないため、こちらで作成する必要がある。

評価は少なくとも以下の 2 種類を分けて行う。

* 正例評価:
  データベースに登録されている論文を正しく同定できるか
* 負例評価:
  データベースに登録されていない文献を誤って同定しないか

### 正例データ

正例データは、JaLC のメタデータから参考文献文字列を生成して作る。
ただし、いきなり誤植付き正例を作るのではなく、まず無加工の正例データを作り、それを複製して誤植付き正例データを作る。

想定内容:

* MongoDB 上の JaLC メタデータから対象論文を抽出する
* 参考文献文字列は以下 3 つの引用スタイルで生成する
  * 情報処理学会
  * 人工知能学会
  * 日本言語学会
* まず誤植なしの正例データを作る
* その後、無加工正例データを複製して誤植付き正例データを作る
* 正解 DOI を保持する

MongoDB の書誌 `_id` は実質連番に近く、かつ `_id` にインデックスがある前提なので、以下の方法を使う。

1. `_id` の最小値・最大値の範囲で乱数を振る
2. 乱数から作った `sampled_id` 以上の `_id` を持つ文書を 1 件取得する

誤植の方針は、論文の表2の確率設定を基本的にそのまま採用する。
具体的には、参考文献文字列ごとに 35% の確率で誤植を発生させる。
誤植を発生させる場合は、80% の確率で 1 個、20% の確率で 2 個の誤植を加える。
書誌要素ごとの誤植発生率と誤植内容は、少なくとも以下で固定する。

* 著者名:
  20%、最終著者の削除
* 論文タイトル:
  40%、1 文字の脱字
* 雑誌名:
  15%、1 文字の脱字
* 出版年:
  5%、元の値 ±1
* 巻:
  5%、元の値 ±1
* 号:
  5%、元の値 ±1
* ページ番号:
  10%、元の値 ±1

ただし、著者が 1 名のみの場合は削除を行わない。
また、巻・号・ページ番号の値が数値でない場合は変更しない。
また、サンプリング結果は再現性確保のため保存して固定する。
生成した無加工正例データも固定ファイルへ保存し、評価や派生データ生成ではその保存済みデータを再利用する。
生成した正例データも固定ファイルへ保存し、評価時にはその保存済みデータを再利用する。

負例データもこちらで作成する。

想定内容:

* 無加工正例データの書誌要素を、論文の表3に沿って改変する
* データベース未登録文献を指す文字列を人工的に作る
* 正解 DOI は持たない

少なくとも以下は固定する。

* 論文タイトル:
  AI による意味を保った書き換え
* 雑誌名:

* 出版年:

### 負例でのフィールド集合

負例では、意図的に `volume` / `page` などの書誌要素を除くケースがありうる。
この場合、RC / CC 計算に使うフィールド集合もその条件に合わせて切り替える。

つまり、

* クエリ側でその要素を使わない
* 候補側でも同じ要素を計算対象から外す

という対称な扱いにする。

### 正例

* 検出漏れ

## 推奨実装ステップ

### 1. 評価データ形式を決める

まず、無加工正例・正例・負例データをどの JSON / dict 形式で持つか決める。

最低限持たせたい情報:

* `reference_text`
* `label` (`positive` / `negative`)
* `doi` または `None`
* `field_names` または評価条件

固定ファイルも段階ごとに分ける。

* `sampled_source_docs.*`
  MongoDB から取得した元メタデータ
* `base_positive_examples.*`
  引用スタイル化だけを行った無加工正例データ
* `positive_examples.*`
  引用スタイル化と誤植付与後の正例データ
* `negative_examples.*`
  改変後の負例データ

### 2. サンプリング済み元データの固定処理を実装する

MongoDB 上の JaLC メタデータから、`_id >= sampled_id` を使ったランダムサンプリングで 50 件程度の論文を取得する。

取得条件には少なくとも以下を含める。

* `content_type = "JA"`
* DOI あり
* 評価に必要な必須書誌要素あり

取得した 50 件は、再現性確保のため固定ファイルへ保存する。

### 3. 無加工正例データ生成を実装する

固定した元メタデータを入力として、AI で以下 3 つの引用スタイルへ整形し、無加工正例データを作る。

* 情報処理学会
* 人工知能学会
* 日本言語学会

最終的に無加工正例データは 150 件程度を想定する。
生成した無加工正例データは固定ファイルへ保存し、以後の評価用データ生成ではその保存済みデータを使う。

### 4. 正例データ生成を実装する

固定した無加工正例データを複製し、論文の表2に従って誤植を付与した正例データを作る。

このとき、論文の表2に従い、35% の確率で誤植を発生させる。
誤植発生時は 80% の確率で 1 個、20% の確率で 2 個の誤植を加える。
生成した正例データは固定ファイルへ保存し、以後の評価ではその保存済みデータを使う。

### 5. 負例データ生成を実装する

固定した無加工正例データを加工して、データベース未登録文献を指す文字列を作る処理を実装する。
改変内容は、論文の表3に従って固定する。
生成した負例データは固定ファイルへ保存し、以後の評価ではその保存済みデータを使う。

### 6. 単一候補への RC / CC / MC 計算を実装する

まず 1 クエリ・1 候補で式どおり計算できる状態を作る。

### 7. 候補群 100 件へのスコア付与と再ランキングを実装する

検索結果に対して RC / CC / MC を付与し、方式別ランキングを作る。

### 8. 閾値判定を実装する

方式ごとの 1 位候補へ閾値を適用し、同定成功 / 失敗を判定する。

### 9. 正例評価・負例評価を実装する

方式ごとに正常検出、検出漏れ、誤検出を集計する。
ここでは 150 件程度の無加工正例データ、正例データ、負例データに対する連続検索と全件評価を担当する。

### 10. notebook で中間確認しながら全体を接続する

最後に、1 件確認用 notebook と全件評価用 notebook を分けたまま、検索 -> 再ランキング -> 閾値判定 -> 評価までを notebook から実行できるようにする。

## 注意点

### 1. トークナイザ不一致を起こさない

登録時、検索時、評価時でトークナイザがずれると、RC / CC / MC の意味が壊れる。

### 2. クエリと候補で同じフィールド集合を使う

負例で `volume` / `page` を除いたなら、候補側でも同じ集合で評価する。

### 3. 閾値と候補件数を分けて考える

`rows=100` は候補取得の幅であり、閾値そのものとは別のパラメータである。

### 4. 評価時は方式ごとに完全に分離する

RC / CC / MC は別スコアなので、順位・閾値・評価結果も別々に管理する。

## 結論

本フェーズでは、

* Solr / BM25 で上位 100 件を取得
* アプリ側で RC / CC / MC を再計算
* 方式別に再ランキング
* 方式別に閾値判定
* 正例・負例データで評価

という流れで実装する。

そのため、次の実装では「評価データ生成」「スコア計算」「再ランキング」「閾値判定」「評価集計」を責務分離して進める。
